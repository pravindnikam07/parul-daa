# Real World Motivation

Imagine you are organizing a cricket tournament.

There are many teams, but only 8 can participate.

How do you choose them?

You don't randomly pick teams.

Instead, you define a strategy.

Example:

- Team should be registered.
- Team should have paid registration fees.
- Team should have at least 11 players.
- Higher ranked teams get priority.

Notice something?

You are making decisions based on **rules**.

Every Greedy algorithm also follows a fixed set of rules.

These rules are called the **Elements of Greedy Strategy**.

# What is Greedy Strategy?

A Greedy Strategy is a systematic process of making the best possible decision at every step while maintaining a valid solution.

Unlike brute force, Greedy does **not** explore every possible solution.

Instead, it follows predefined rules.

---

No Candidate Set

Nothing to choose.

Impossible.

---

No Selection Function

Cannot decide.

---

No Feasibility Function

May produce invalid answer.

---

No Objective Function

Don't know what to optimize.

---

No Solution Function

Algorithm never stops.

# Difference Between Greedy Strategy and Greedy Algorithm

| Greedy Strategy               | Greedy Algorithm        |
| ----------------------------- | ----------------------- |
| Planning phase                | Implementation phase    |
| Defines how to make decisions | Executes the strategy   |
| Concept                       | Program                 |
| Independent of language       | Implemented in Java/C++ |

Think of **Strategy** as the blueprint and **Algorithm** as the actual building.

---

# Java Example

This example demonstrates the **Selection Function** and **Feasibility Function** using the coin change problem.

```java
public class GreedyElements {

    public static void main(String[] args) {

        int[] coins = {50, 20, 10, 5, 2, 1};
        int amount = 87;

        System.out.println("Selected Coins:");

        for (int coin : coins) {

            // Feasibility Check
            while (amount >= coin) {

                // Selection Function
                System.out.print(coin + " ");

                amount -= coin;
            }
        }
    }
}
```

---

# C++ Example


# Time Complexity

The overall complexity depends on the implementation:

| Operation                  | Complexity             |
| -------------------------- | ---------------------- |
| Linear candidate selection | O(n)                   |
| Sorted candidates          | O(n log n) (sorting)   |
| Priority Queue selection   | O(log n) per selection |

# Advantages of Greedy Strategy

- Easy to understand and implement.
- Fast execution.
- Low memory usage.
- Suitable for many optimization problems.
- Often provides optimal solutions for problems satisfying greedy properties.

---

# Limitations

- Does not always produce the optimal solution.
- Irreversible decisions.
- Requires proof of correctness.
- Not suitable for all optimization problems.

---

# Common Mistakes

- Confusing Selection Function with Objective Function.
- Ignoring Feasibility Checks.
- Assuming Greedy always works.
- Not identifying the correct Candidate Set.
- Forgetting the stopping condition.

---

# Best Practices

- Clearly identify all five elements before coding.
- Verify the **Greedy Choice Property** and **Optimal Substructure**.
- Use sorting or priority queues when they simplify selection.
- Test with edge cases and counterexamples.
- Keep the selection criterion simple and deterministic.

# Summary

- A **Greedy Strategy** is a decision-making framework that builds a solution by repeatedly choosing the best available option.
- Every Greedy algorithm is built around **five essential elements**:

  1. Candidate Set
  2. Selection Function
  3. Feasibility Function
  4. Objective Function
  5. Solution Function

- Separating these elements makes algorithm design easier and helps in proving correctness.
- Understanding these concepts is essential before studying specific Greedy algorithms such as **Kruskal's**, **Prim's**, **Dijkstra's**, **Activity Selection**, **Fractional Knapsack**, and **Huffman Coding**, which are the remaining topics in this unit.
